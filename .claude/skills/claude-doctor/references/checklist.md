# `.claude/` 監査チェックリスト

`/claude-doctor` スキルが監査委譲時に **そのままプロンプトに展開**する詳細仕様。観点・参照ドキュメント・出力フォーマット・必守ルールを単一ファイルで定義し、スキル本体は薄いフロー制御に専念する。

---

## 監査の観点（カテゴリ別）

### A. settings.json
- `hooks` の網羅性: PreToolUse / PostToolUse / UserPromptSubmit / SessionStart / Stop など、活用すべきイベントを使えているか
- `permissions` の整理: allow/deny の粒度、過剰許可がないか
- `env`、`statusLine`、`outputStyle` の活用余地
- 相対パス vs `$CLAUDE_PROJECT_DIR` の統一感
- timeout 指定の妥当性

### B. agents/（サブエージェント）
- frontmatter の `description` が **1 行で書かれているか**（複数行だと全エージェント未読込になる既知の落とし穴）
- `tools` の絞り込み（過剰な権限付与になっていないか）
- `model` 指定の妥当性（重い分析なら opus、軽量なら haiku/sonnet）
- 各エージェントの役割が重複していないか
- 命名: kebab-case か

### C. commands/（スラッシュコマンド）
- frontmatter の `description` / `allowed-tools` / `argument-hint` の充足
- `user-invocable` / `disable-model-invocation` の使い分け
- 引数（`$ARGUMENTS`）の扱いが明示されているか
- 既存スキル/エージェントとの責務分担が明確か

### D. hooks/（フックスクリプト）
- スクリプトに実行権限（chmod +x）があるか
- `INPUT="$(cat)"` パターンで stdin を一度キャプチャしているか（多重 read を避ける）
- jq / python3 フォールバックがあるか
- 終了コードの使い分け（0=許可 / 2=ブロック / 1=エラー）が適切か
- セルフトリガー（フック自身を編集する際にブロックされる）の除外が入っているか
- timeout 指定の有無

### E. skills/（スキル）
- 各 `SKILL.md` の `description` が起動判定として十分か
  - **Runmates スタイル**: 日本語で 150〜300 文字、トリガー語を「」で囲む、暗黙文脈の救済、優先関係を含む
  - 「Helps with X」のような曖昧表現は NG
- progressive disclosure: 500 行超なら `references/` / `scripts/` / `assets/` への分割を提案
- `paths:` フィルタの活用余地（特定パス編集時のみロード）
- `user-invocable` / `disable-model-invocation` の使い分け
- 公式 marketplace / `npx skills find` で代替できるものを自作していないか

### F. rules/（横断ルール）
- `paths:` 指定で自動ロード対象を絞れているか
- CLAUDE.md からの参照リンクが切れていないか
- 内容の重複（複数ファイルで同じことを書いていないか）

### G. CLAUDE.md / MEMORY.md
- CLAUDE.md の鉄則がプロジェクトの実態と一致しているか
- 参照されている `.claude/rules/` ファイルが実在するか
- MEMORY.md のエントリと現在のコードベースに矛盾がないか

### H. セキュリティ
- secret 検出・破壊コマンド防止のガードがあるか（無ければフック追加を提案）
- `.gitignore` で `.claude/settings.local.json` などローカル設定が除外されているか
- API キー・接続文字列の混入リスク
- ハードコードされた絶対パス（`/Users/...`）の混入チェック（共有性を損なう）

## 進め方

1. **スコープ確認**: 呼び出し元から受け取った `scope` / `repository_path` を確認。指定が無ければ `.claude/` 全体・カレントリポジトリを対象とする
2. **現状把握**: `Glob` / `Read` で `.claude/` 配下を一通り読む。大きすぎる場合はカテゴリ別に絞る
3. **公式仕様の確認**: 必要に応じて `WebFetch` で最新ドキュメントを取得（並列化推奨）
4. **観点別に評価**: 上記カテゴリ A〜H を順に評価。該当しないカテゴリはスキップ可
5. **優先度分類**: 各指摘を 🔴（必修正・実害あり）/ 🟡（改善推奨）/ 🟢（良い点）に分類
6. **修正案を添える**: 「何が問題か」+「具体的にどう直すか」をセットで。可能ならコード片を示す
7. **次週への宿題**: トップ 3〜5 件のアクション項目を抽出する

## 優先度の分類基準（ブレ防止）

🔴/🟡/🟢 は感覚で振らず、以下の基準で機械的に判定する。判断に迷う場合は **一段下げる**（過剰な🔴連発を避ける）。

### 🔴（必修正・今週中に対応）
以下のいずれかに該当する場合のみ：
- **実害発生中**: 既に動作不良・誤検知ブロック・誤通過を引き起こしている（再現手順を書ける）
- **公式仕様違反**: WebFetch で取得した最新ドキュメントの「必須項目」「禁止事項」に違反している（URL と該当箇所を併記）
- **セキュリティリスク**: 秘密情報の漏洩可能性・破壊的コマンドの素通り経路がある
- **CLAUDE.md / MEMORY.md 規約違反**: プロジェクトの明文化された鉄則に反する記述・実装がある

→ 1 カテゴリにつき 0〜2 件を目安。3 件以上挙がる場合は本当に全部🔴か再評価する。

### 🟡（改善推奨・今月中に検討）
以下のいずれかに該当：
- ベストプラクティスから外れているが現状の動作に支障なし
- 将来の保守性・拡張性を損なう設計選択
- 公式ドキュメントで「推奨」扱いの項目を採用していない

→ 1 カテゴリにつき 3 件まで。それを超えるものは「要確認」枠か🟢に統合する。

### 🟢（良い点・現状維持）
- 公式仕様・プロジェクト規約に準拠し、かつ非自明な設計判断
- 「単に動いている」だけのものは🟢に書かない（ノイズになる）

→ 1 カテゴリにつき 2〜4 件。多すぎると褒め一覧になりレポートの価値が落ちる。

### 「要確認」枠
- 実物を grep / Read で確認できなかった項目
- WebFetch が失敗した参照ドキュメント
- 推測が混じる指摘

→ 🔴/🟡 に分類せず「**要確認**: 〇〇」と明記。断定しない。

## 出力前セルフチェック（必須）

レポートをユーザーに返す前に以下を **自分で順に確認**し、引っかかったら出力前に直す：

- [ ] **観点 A〜H を全て触れたか**（該当なしのカテゴリは「該当なし」と明記、無言で飛ばさない）
- [ ] **全ての🔴と🟡に `file:line` 形式の参照があるか**（行番号無しはセルフチェック失格）
- [ ] **全ての🔴と🟡に修正案があるか**（「問題のみ指摘して修正案無し」は禁止）
- [ ] **🔴 の根拠が「実害・公式違反・セキュリティ・規約違反」のどれに該当するか自分で言えるか**
- [ ] **WebFetch で公式 docs を取得したか**（取得した URL を「参照した公式ドキュメント」セクションに列挙）
- [ ] **絶対パス（`/Users/...`）がレポート本文に混入していないか**（grep で自己チェック）
- [ ] **「要確認」枠の項目は断定形で書かれていないか**
- [ ] **次週の宿題はトップ 3〜5 件に絞られているか**（10 件並べない）

セルフチェックに通らない場合、ユーザーに出す前に該当箇所を修正する。

## 必守ルール

- **修正は行わない**。あくまで発見と提案のみ（修正は呼び出し元スキルが別フローで処理する）
- 指摘には **必ず file:line と修正案** を添える
- 確証のない指摘は「要確認」と明示し、断定しない
- 「思いつき」の追加機能提案ではなく、ベストプラクティス準拠と運用性に絞る
- lazy-skeptic 観点で「本当に必要か？」を自問してから提案する（過剰な機能追加を勧めない）
- 公式 marketplace に既存スキルがある機能は、自作ではなくインストールを提案する
- 絶対パス（`/Users/...`）をレポートに書かない（`$CLAUDE_PROJECT_DIR` / 相対パスを使う）
- 報告は構造化して、後から呼び出し元スキルが整形しやすい形にする
